## PR 1 — feature/membership-provisioning (C.9)

### Goal

When a circle wants to onboard a new member, the Clerk handles the technical provisioning: Authentik account → Loomio group membership → Mattermost welcome. The member arrives ready to participate without manual admin steps.

**Out of scope for this PR:** member application, circle vote to admit, ZK proof-of-identity. These are Phase B (noted for the full C→ lifecycle path, with zero-knowledge identity considered).

### New service: `src/IskanderOS/services/provisioner/`

A thin FastAPI service. Follows the same structure as `decision-recorder`: Dockerfile, `requirements.txt`, `main.py`, `db.py` (for provisioning audit log).

**Endpoints:**

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/members` | Provision a new member across all three systems |
| `GET` | `/members/{username}` | Check provisioning status |
| `GET` | `/health` | Service health |

**`GET /members/{username}` response:** `{username, authentik_exists: bool, loomio_member: bool, mattermost_notified: bool, provisioned_at: ISO8601|null}`. Queries the provisioning_log table by username. Returns 404 if username has no provisioning record. Useful for re-run idempotency checks — if `authentik_exists` is true but `loomio_member` is false, the operator knows exactly where to resume.

**`POST /members` flow:**
1. Validate request: `{username, email, display_name}` — Pydantic model, length limits
2. Call Authentik admin API (`/api/v3/core/users/`) — create account, set `is_active=true`
3. Generate Authentik one-time recovery link — use `/api/v3/core/users/{id}/recovery/` (generates an expiring single-use link the user follows to set their own password). **Do NOT use `/set_password/` — that sets the password to a caller-supplied value and would require transmitting a known password in the welcome message.** The exact endpoint must be confirmed against the deployed Authentik version before implementation; the Authentik changelog for the target version should be checked.
4. Call Loomio API — add user to the cooperative's default group via `/api/v1/memberships`
5. Post welcome message to a **dedicated onboarding channel** (not `#general`) — includes password reset link, pointer to Clerk, first steps. The password reset link must not be posted to a world-readable channel; the operator guide must document setting up a restricted `#onboarding` channel and configuring `PROVISIONER_WELCOME_CHANNEL_ID`.
6. Write audit record to PostgreSQL (provisioning_log table: username, email, provisioned_at, authentik_id, loomio_membership_id, mattermost_post_id)
7. Return `{username, authentik_id, loomio_membership_id, mattermost_post_id, password_reset_url}`

**Auth:** `AUTHENTIK_TOKEN` (bootstrap token, already generated by installer). The operator guide must note that this is a permanent superuser-scoped token; Phase B should replace it with a scoped Authentik service account. Protected by `_verify_internal_caller` pattern (same as decision-recorder). NetworkPolicy restricts callers to openclaw + Traefik only.

**Rate limiting:** sliding window, same pattern as other services.

### Clerk extensions

New tool: `provision_member` — **write tool** (Glass Box required before calling).

```
provision_member(username, email, display_name) → provisioner POST /members
```

`SOUL.md` additions — "New member onboarding" section:
- Clerk collects name and email from the requesting member
- Confirms the details before provisioning ("I'm about to create an account for X with email Y — shall I go ahead?")
- Calls `glass_box_log` then `provision_member`
- Reads the welcome message back into the channel so the requesting member sees what was sent

`agent.py`: `provision_member` added to `_WRITE_TOOLS`. **`provision_member` is NOT added to the actor-injection check** — the LLM supplies `username`, `email`, and `display_name` as explicit tool arguments for the new member being created. `actor_user_id` (the requesting member's Mattermost ID) is only injected for Glass Box logging and for tools where the actor and the target are the same person. Adding it to `provision_member` would cause a double-injection where the requesting member's ID overwrites the new member's username.

### Helm chart

`infra/helm/iskander/templates/provisioner.yaml` — same pattern as `decision-recorder.yaml`:
- Deployment + Service + NetworkPolicy
- Secret refs: `AUTHENTIK_TOKEN`, `LOOMIO_API_KEY`, `MATTERMOST_BOT_TOKEN`, `LOOMIO_DEFAULT_GROUP_KEY`
- `values.yaml` additions: `provisioner.enabled`, `provisioner.loomioDefaultGroupKey`
- Installer `generated-values.yaml.j2` already generates `AUTHENTIK_TOKEN`; add `LOOMIO_DEFAULT_GROUP_KEY` prompt to first-boot wizard

### Documentation

- `docs/operations/member-provisioning.md` — operator guide: how provisioning works, what secrets are needed, how to re-run if it fails mid-way
- Inline docstrings on all public functions

## PR 2 — feature/meeting-prep-clerk (#47)

### Goal

The Clerk can assemble a structured meeting agenda on request, pulling live data from the decision recorder and Loomio. The facilitator gets a ready-to-share document without manual data gathering.

### New Clerk tools (all read-only — no Glass Box required)

**`prepare_meeting_agenda(circle_key, meeting_type)`**

Assembles from live sources:
- Open tensions for the circle (`GET /tensions?status=open&domain={circle_key}`)
- Agreements due for review in next 14 days (`GET /decisions/reviews/due?days_ahead=14`)
- Proposals closing in the next 7 days (`loomio_list_proposals(group_key)` filtered by closing date)
- Last 3 recorded decisions for context (`GET /decisions?group_key={circle_key}&limit=3`)

Returns a structured markdown agenda using the cooperative's standard format:
```
## [Meeting type] — [Circle] — [Date]

### Check-in (10 min)

### Administrative consent (5 min)
- Approve previous minutes

### Tensions to process
[list from tension tracker]

### Proposals for consent
[list from Loomio]

### Agreements due for review
[list from decision recorder]

### Any other business

### Check-out (5 min)
```

`meeting_type` options: `circle`, `agm`, `board`, `working_group`

**`list_recent_decisions(group_key, limit)`**

Meeting-friendly summary of the last N decisions: title, outcome sentence, decided date. Wraps existing `GET /decisions` — exposed as a distinct Clerk tool for discoverability.

**`draft_meeting_notice(circle_key, date, facilitator)`**

Local formatting only (no API call). Produces a Mattermost-ready meeting announcement with date, facilitator, and a link to the Loomio group. The Loomio group URL is constructed as `{LOOMIO_BASE}/g/{circle_key}` — the same pattern used in `loomio_create_discussion`. Members can post it themselves or ask the Clerk to post it (which then uses `mattermost_post_message` — an existing write tool that already requires Glass Box).

### `SOUL.md` additions — "Meeting facilitation" section

- Role rotation reminder: if the same facilitator has run the last 3 meetings, Clerk notes this and suggests rotation (S3: role diversity)
- Quorum: Clerk reminds the facilitator how many members are needed for consent decisions
- Consent agenda format: explain that administrative items go under "administrative consent" and are approved as a block unless someone pulls an item
- Clerk never chairs the meeting — it prepares and advises

### Documentation

- `docs/operations/meeting-prep.md` — guide for facilitators: what the Clerk can prepare, how to request an agenda, what meeting types are available
- Governance template `docs/templates/governance/meeting-agenda.md` — the blank agenda format for reference

## PR 3 — feature/e2e-verification (C.10)

### Goal

Any contributor (or the cooperative itself) can run a single script against a deployed Debian VM and get a clear pass/fail on the entire Phase C stack. CI runs the same script on every PR.

### `verify.sh`

Location: `verify/verify.sh`  
Input: `ISKANDER_HOST` environment variable (defaults to `localhost`)  
Output: structured log with pass/fail per check, exit code signals failure tier

**Tier 1 — Infrastructure (exit 1 on failure)**
- K3s node in Ready state: `kubectl get nodes`
- All expected pods Running or Completed (no CrashLoopBackOff, no Pending > 2 min)
- Expected pods: authentik, loomio, mattermost, nextcloud, postgresql, redis, ipfs, decision-recorder, openclaw, provisioner, vaultwarden, backrest, beszel

**Tier 2 — Service health (exit 2 on failure)**
- `GET /health` on each service that exposes it: decision-recorder, openclaw, provisioner
- Authentik: `GET /api/v3/-/` (liveness)
- Loomio: `GET /api/v1/groups` (authenticated)
- Mattermost: `GET /api/v4/system/ping`
- Nextcloud: `GET /status.php`
- IPFS: `GET /api/v0/id`

**Tier 3 — Journey smoke test (exit 3 on failure)**
1. Provision test member: `POST /members` via provisioner with `verify-{UUID}` username → verify `GET /members/verify-{UUID}` returns `authentik_exists: true, loomio_member: true, mattermost_notified: true`
2. Clerk interaction: send test message to openclaw → verify non-empty response
3. Glass Box write: call `POST /log` → verify entry readable via `GET /log?actor=verify-{UUID}`
4. Glass Box enforcement (unit test, not smoke test): the `_WRITE_TOOLS` ordering guard operates inside the agent loop — it cannot be tested via a plain HTTP call to the provisioner. This invariant is covered by `tests/test_agent.py::test_write_tool_without_glass_box_rejected`, which calls `_validate_response_ordering()` directly with a fabricated response block containing `provision_member` without `glass_box_log` and asserts it returns an error string. The smoke test step 4 instead verifies the happy path: the `glass_box_log` entry created in step 3 is readable via `GET /log` and its `action` field contains the correct value.
5. Decision recording: send mock webhook to `POST /webhook/loomio` (HMAC-signed with test secret) → verify decision appears in `GET /decisions`
6. Loomio member check: verify the provisioned `verify-{UUID}` user appears in `GET /api/v1/memberships` for the default group (not a discussion — `provision_member` does not create discussions)
7. Cleanup: call Authentik admin API to delete `verify-{UUID}` user; Loomio membership is cascade-deleted with the user; verify `GET /members/verify-{UUID}` returns 404 after cleanup

All test resources use the same `verify-{UUID}` prefix. Because UUIDs are generated fresh per run, re-runs cannot collide. The cleanup step is idempotent — if the UUID user does not exist, the delete call is a no-op.

**CI approach for external-API steps:** `values-ci.yaml` configures the provisioner to point at in-cluster Authentik and Loomio (both deployed in the K3d cluster). No real external API keys are needed in CI — all services run locally within the K3d cluster. CI secrets (`ANTHROPIC_API_KEY` for openclaw) are passed as GitHub Actions secrets and injected into `values-ci.yaml` via `--set` flags.

All test resources are prefixed `verify-{UUID}` to avoid collisions with real data.

**Exit codes:**
| Code | Meaning |
|------|---------|
| 0 | All tiers passed |
| 1 | Infrastructure not ready |
| 2 | One or more services unhealthy |
| 3 | Journey smoke test failed |

**CI-friendly output:** `::error::` and `::notice::` annotations for GitHub Actions. Pass/fail summary table at the end.

### `.github/workflows/verify.yml`

Triggers: push to `main`, pull requests targeting `main`

Steps:
1. Install K3d + kubectl + Helm
2. Create K3d cluster (`k3d cluster create iskander-ci --agents 1`)
3. Generate test secrets (`verify/generate-ci-secrets.sh` — deterministic values for CI, not for production)
4. `helm upgrade --install iskander ./infra/helm/iskander -f verify/values-ci.yaml`
5. Wait for all pods ready (`kubectl wait --for=condition=ready pod --all -n iskander --timeout=300s`)
6. `ISKANDER_HOST=localhost bash verify/verify.sh`
7. Upload verify log as workflow artifact

### `verify/values-ci.yaml`

Minimal Helm overrides for CI: reduced resource requests, ephemeral storage, CI-safe credentials. Committed to the repo (no real secrets).

### Documentation

- `docs/operations/verification.md` — operator guide: how to run `verify.sh` against your Debian VM, what each tier checks, how to interpret failures
- `verify/README.md` — quick reference for contributors running verification locally

## Constraints and coordination

- **Steward agent (C.6):** being built by another contributor. This sprint does not touch `openclaw/agents/steward/`. If the Steward contributor needs to extend `tools.py` or `agent.py`, coordinate via #53.
- **PR #49 (S3 governance):** should merge before `feature/membership-provisioning` branches to avoid conflicts on `tools.py`.
- **`decision-recorder` db.py:** the provisioner uses its own PostgreSQL database/table. It does not extend `decision-recorder`'s schema to avoid conflicts.
- **Phase B consideration:** the full membership lifecycle (application → circle vote → ZK proof-of-identity) is explicitly out of scope. The provisioner's `POST /members` network path is compatible with a future vote-outcome webhook from Loomio (the Loomio pod is an allowed NetworkPolicy ingress source). Payload translation will be required — a Loomio outcome webhook has a different shape to `{username, email, display_name}`. Phase B adds a translation handler; `POST /members` itself does not need structural changes.
