# Member Provisioning — Operator Guide

## 1. Overview

The provisioner is a FastAPI service that onboards new cooperative members across three systems: Authentik (SSO identity), Loomio (governance group membership), and Mattermost (welcome notification). It is called exclusively by the Clerk agent (OpenClaw); no other service or operator interacts with it directly. The provisioner is idempotent — if the Clerk retries a failed request, already-completed steps are skipped.

---

## 2. Prerequisites

### Authentik — service account and API token

1. Log in to your Authentik admin interface.
2. Go to **Directory > Service Accounts** and create a new service account. Name it `provisioner-bot` or similar.
3. On the service account's detail page, select **Create Token**. Set the token's expiry to "never" (or rotate it on your security schedule).
4. The token needs permission to call the core users admin API (`POST /api/v3/core/users/` and `POST /api/v3/core/users/{id}/recovery/`). Assign the service account to the `authentik Admins` group, or create a custom permission group scoped to the Users model.
5. Copy the token value — you will only see it once.

> This must be a service account token, not a personal API key. Personal tokens are tied to your admin user and will break if that user's password changes or the account is locked.

### Loomio — group key

1. Log in to Loomio as a coordinator of your cooperative's group.
2. Open the group settings page. The group key is the short identifier in the URL: `https://loomio.example.coop/g/<GROUP_KEY>/`.
3. Note this value — you will set it in `values.yaml`.

### Mattermost — onboarding channel

1. In Mattermost, create a private channel named something like `#new-members`. Do not use `#general` or any public channel.
2. Add the Iskander bot account to this channel (it must be a member, not just an admin).
3. Find the channel ID: open the channel in Mattermost, go to **Channel Info > Edit**, and copy the ID from the URL — it looks like `abc123def456`.

### Provisioner database

The provisioner needs a dedicated PostgreSQL database and role. The first-boot install script creates these automatically. If you are setting up manually, run:

```sql
CREATE ROLE provisioner WITH LOGIN PASSWORD '<strong-password>';
CREATE DATABASE provisioner OWNER provisioner;
```

## 3. Configuration (`values.yaml`)

The `provisioner:` block in `values.yaml`:

```yaml
provisioner:
  enabled: true          # Set to false to disable the provisioner entirely
  image:
    repository: ghcr.io/argocyte/provisioner
    tag: "0.1.0"         # Pin this; update intentionally after testing
  logLevel: "INFO"       # Set to "DEBUG" when diagnosing problems

  postgresql:
    password: ""         # Generated by the first-boot script; do not set manually
                         # unless you initialised the DB yourself (see Section 2)

  authentik:
    apiToken: ""         # Service account token with admin user scope (see Section 2)
                         # MUST be a service account token — not a personal admin token

  # The short group identifier from your Loomio group URL:
  # https://loomio.example.coop/g/<THIS_VALUE>/
  loomioGroupKey: ""

  # Channel ID of the restricted Mattermost onboarding channel.
  # Copy from the channel URL in the Mattermost web app.
  # Must NOT be a public channel like #general.
  mattermostOnboardingChannel: ""

  resources:
    requests:
      memory: "64Mi"
      cpu: "25m"
    limits:
      memory: "128Mi"
      cpu: "200m"
```

The Loomio API key and Mattermost bot token are pulled from the `openclaw:` block and shared with the Clerk agent — you do not set them again here.

The optional `decisionRecorder.internalServiceToken` value, if set, is used to add bearer-token authentication on the provisioner's HTTP interface on top of the Kubernetes NetworkPolicy. Without it, NetworkPolicy alone restricts access to pods labelled `openclaw`.

## 4. First-time setup

### Step 1 — Create Authentik service account

Follow the Authentik steps in Section 2. Copy the token.

### Step 2 — Create Loomio group and note the key

If the cooperative's Loomio group already exists, just note its key from the URL. If not, create the group as a Loomio coordinator, then note the key.

### Step 3 — Create the Mattermost onboarding channel

Create the private `#new-members` channel. Add the bot. Copy the channel ID from the Mattermost URL.

### Step 4 — Set values in `values.yaml`

```yaml
provisioner:
  authentik:
    apiToken: "eyJ..."               # Authentik service account token
  loomioGroupKey: "oursunrise"       # Your group's key from the Loomio URL
  mattermostOnboardingChannel: "abc123def456"  # Mattermost channel ID
```

### Step 5 — Deploy

```bash
helm upgrade --install iskander ./infra/helm/iskander -f values.yaml
```

Verify the provisioner started:

```bash
kubectl get pods -n iskander -l app.kubernetes.io/component=provisioner
kubectl logs -n iskander -l app.kubernetes.io/component=provisioner --tail=30
```

The log should end with `INFO provisioner Provisioner ready.`

## 5. How member provisioning works

When the Clerk triggers a provisioning request, the service runs three steps in order:

**Step A — Authentik.** The provisioner calls `POST /api/v3/core/users/` to create an SSO account. It then immediately calls `POST /api/v3/core/users/{id}/recovery/` to generate a single-use, time-limited password-set URL. This URL is stored in the provisioner database and returned to the Clerk. The provisioner never sets a password — the member sets their own password by following the link.

**Step B — Loomio.** The provisioner calls `POST /api/v1/memberships` to add the member's email address to the cooperative's Loomio governance group. The membership ID is stored.

**Step C — Mattermost.** The provisioner posts a welcome message to the onboarding channel, directed to the new member by display name and username.

**Idempotency.** After each step succeeds, its result is written to the provisioner's PostgreSQL database. If a step fails and returns a 502, the record already contains the results of any steps that completed. On retry, the provisioner checks the record and skips any step already marked complete. The Clerk can safely retry without double-creating accounts.

The password reset URL is a single-use Authentik recovery link. It is not a stored password. If the link expires before the member uses it, an operator can generate a new one from the Authentik admin interface under the member's account.

## 6. Triggering via the Clerk

Send the Clerk a Mattermost message in any channel it monitors:

```
@clerk please provision a new member:
- Username: alice
- Email: alice@oursunrise.coop
- Display name: Alice
```

Username must be lowercase letters, digits, hyphens, or underscores only.

The Clerk will read back the request and ask you to confirm before it acts (Glass Box requirement — the Clerk never performs provisioning without explicit human confirmation). After you confirm, the Clerk calls `POST /members` on the provisioner and relays the result, including the password reset URL for the new member.

## 7. Checking provisioning status

From inside the cluster (or via `kubectl exec` into any pod in the `iskander` namespace):

```bash
curl http://provisioner.iskander.svc.cluster.local:3001/members/alice
```

Example response:

```json
{
  "username": "alice",
  "email": "alice@oursunrise.coop",
  "display_name": "Alice",
  "authentik_id": "a1b2c3d4-...",
  "loomio_membership_id": 42,
  "mattermost_post_id": "xyz789",
  "password_reset_url": "https://auth.oursunrise.coop/...",
  "provisioned": true,
  "authentik_exists": true,
  "loomio_member": true,
  "mattermost_notified": true
}
```

**Flag meanings:**

| Flag | Meaning |
|---|---|
| `authentik_exists` | SSO account created in Authentik |
| `loomio_member` | Email added to the cooperative's Loomio group |
| `mattermost_notified` | Welcome post sent to the onboarding channel |
| `provisioned` | All three steps complete |

If `provisioned` is `false`, one or more of the individual flags will be `false`. That tells you which step failed. Retry is safe.

## 8. Troubleshooting

### 502 from the provisioner

The provisioner returned 502 because one step failed, but earlier steps may have succeeded. The partial state is already saved. Check which step failed in the log:

```bash
kubectl logs -n iskander -l app.kubernetes.io/component=provisioner --tail=50
```

Fix the underlying cause (see below), then retry the provisioning request. The provisioner will skip completed steps automatically.

### Authentik 403

The API token does not have permission to create users or generate recovery links. Verify the service account is in the `authentik Admins` group (or has explicit Users model permissions). Rotate the token in Authentik, update `provisioner.authentik.apiToken` in `values.yaml`, and redeploy.

### Loomio 404

The group key is wrong or the Loomio group has been renamed. Check the Loomio group URL and update `provisioner.loomioGroupKey` in `values.yaml`. Redeploy.

### Mattermost 403

The bot account is not a member of the onboarding channel. In Mattermost, open the channel, go to **Members**, and add the bot. No redeploy needed — the fix takes effect on the next retry.

## 9. Phase B notes

This provisioner implements Tier B onboarding: it gives a new member immediate access to the cooperative's SSO, governance forum, and chat without requiring any cryptographic identity proof. Phase B will add a ZK identity layer — members will be able to prove their uniqueness and eligibility using BrightID verification or Soulbound Tokens, with selective disclosure via zero-knowledge proofs so that only what is necessary is revealed. The Tier B flow described in this document will remain in place and will be extended, not replaced — the goal is that existing members do not need to re-provision when the cryptographic layer is deployed.
